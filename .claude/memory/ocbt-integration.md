Integrating the OCBT (pool-based Concurrent Binary Tree, Benyoub & Dupuy HPG 2024) into World42 to remove the implicit GPU CBT's O(2^D) reduction ceiling (~depth 25) and reach **depth ~60** (paper parity). The CBT becomes a fixed-capacity memory-pool allocator; triangulation is stored as explicit bisectors, decoupling cost/memory from subdivision depth.

**KEY Phase 1 finding:** World42's `cbt_state.ts` forced-diamond split (base-only Rivara forcing, which `ocbt_topology.ts` ports verbatim) is watertight + symmetric ONLY for SINGLE-region coherent (metric-driven) refinement — the realistic LOD case. Under multi-region / arbitrary refinement it strands T-junctions (proven: cbt_state and the mirror produce identical violation counts under the same sequence — a property of base-only forcing, not a bug). General watertightness REQUIRES the longest-edge PROPAGATION chain (the reference `update_utilities.hlsl` PropagateBisect / World42's own `cbt_conform.wgsl` splitConforming, which `cbt_conformity.test.ts` validates). That propagation is the headline Phase 1c deliverable. Also deferred to Phase 1c: maxDepth-boundary conformance (refuse a split whose forced closure can't complete — the reference's atomic memory reservation), the concurrent 4-pattern batch engine, the WGSL port, and the GPU↔mirror cross-check.

**Phase 1c-i DONE (2026-06-23): general watertightness via Rivara LEPP.** `ocbt_topology.ts` forceSplit now does the Longest-Edge Propagation Path: while the BASE (split-edge) neighbour is strictly COARSER, recursively forceSplit it and refetch, then bisectDiamond with the same-level partner. This makes it watertight for ANY refinement (multi-region, arbitrary-random, pole, seams) — the depth fixpoint (level test, not the old topology test) is the fix. 48 OCBT tests inc. an adversarial stress suite (`ocbt_topology_stress.test.ts`: deep random crossing >=2 pool grows, pole/seam, split+merge churn, 20 deep seeds) all assert 0 T-junction + symmetry + heapID↔LEB + finite verts.

**Two important bugs found & their lesson:**
1. The "cracks" were NOT the algorithm — they were a **grow()-stale-reference bug**: `subdivide` cached `const v = this.verts` BEFORE `allocSlot()`, which can `grow()` (double the typed arrays) and REPLACE `this.verts`; child verts were written to the orphaned old array → new slots got 0/NaN verts → false T-junctions past 4096 leaves. Fix: read parent verts first, alloc, then re-fetch `this.verts`. **`src/systems/lod/cbt/cbt_state.ts` has the SAME latent bug** (`const v = this.verts` before `allocSlot` in its `subdivide`) — it only triggers above ~4096 live leaves; flag to fix in the production CPU path.
2. Test quantization: the watertight check must quantize verts to a grid (~1e-7 on the unit sphere) coarser than FP accumulation but finer than edges; below ~depth 20 it is reliable, deeper the CHECK (not the topology) gets noisy.

**SHARPER DIAGNOSIS (2026-06-24, end of session) — the root cause is the SEED↔reference cross-link convention, NOT just labels.** Two findings narrowed it down:
1. The GPU heap-id geometry decoder (reference leb.hlsl matrix, spherical midpoints) is correct ONLY when seeded `v0=face.right, v1=face.apex, v2=face.left` (NOT l,a,r). Derived from the seed lane constraint: n2=BASE across (left,right) and n0=LEFT across (apex,left) forces v0=r,v1=a,v2=l. With that seed, refLebDecode reproduces the currentID-chain geometry exactly (verified by hand for h=16 and h=32 in face 0). This decoder lives in ocbt_topology_gpu_test_main.ts `gpuCorners()`.
2. EVEN with the correct decoder, uniform **L1** (just the 4 base diamonds) fails the watertight check: slot 0 (h=16 = face-0's [1,0,0]-side half, seam edge ([1,0,0],midpoint)) is cross-linked across the 0-4 equator to slot 12 (h=25 = face-4's [0,0,1]-side half). Those two halves are on OPPOSITE ends of the shared seam — they meet only at the midpoint, so they are NOT edge-adjacent. The correct seam neighbor of slot 0 is face-4's OTHER child (slot-4-reused, h=24), not slot-4's new sibling (h=25). So `evaluate_neighbors` (BisectElement CENTER cross-link, via `ev.x = idx(twin,0)`) returns the WRONG twin-child for the diamond. The mesh is still a closed manifold (reciprocity + count hold) because the twist is mutually consistent — but it is geometrically twisted at every seam.

CONCLUSION: the reference's `evaluate_neighbors` + the BisectElement neighbor cross-links assume the reference's OWN octahedron seed orientation/winding; World42's seed (lebFaceCorners + ROOT_NEIGHBORS, reference order remap) does not match it, so the twin-child selection is mirrored at seams. This is the "Mapping voisins seed (Prev/Next/Twin ↔ ROOT_NEIGHBORS)" HIGH risk in the plan.

**Phase 3 (PRECISION) — df64 camera-relative decode DONE + validated in-app (2026-06-24).** Lifts the f32 depth ceiling. CRITICAL realization: World42's `doublepos` is a Babylon `Vector3` = **f32**, not real f64 — the whole floating origin is f32. So the ~depth-24 ceiling has two f32 sources: (a) the ~depth-many normalize steps in the unit-dir decode accumulate f32 error; (b) the per-vertex planet-local position `dir*radius` (~radius magnitude) is f32-quantized (ULP ~0.24 km at radius=2439 sim units → adjacent fine vertices snap → cracks at L24+). FIX (both): decode the leaf in **df64** (ocbt_f64.wgsl, ~48-bit) AND emit each corner **camera-relative**: `relative = dir*radius − camLocal` computed in df64, narrowed to f32. The subtraction cancels the ~radius magnitude in df64 so the f32 output is small (near camera) and finely resolved. camLocal is only f32-precise, but its error is a CONSTANT per-frame translation of the whole patch (same camLocal for every vertex) — it shifts the patch uniformly (invisible), it does NOT jitter adjacent vertices, so it cannot crack. The per-vertex precision now comes from the df64 `dir*radius` → crack-free to ~depth 48.
